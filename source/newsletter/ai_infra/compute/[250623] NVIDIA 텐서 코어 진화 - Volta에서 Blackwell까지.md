## 🔑 용어 정리

본문을 순서대로 읽기 전에 알아두면 좋은 용어들입니다. 자세한 수치와 설명은 본문에서 처음 등장하는 위치에 나옵니다.

- **텐서 코어 (Tensor Core)**: GPU 안에서 행렬 곱셈(딥러닝 연산의 핵심)만 전담하는 별도의 전용 회로. 범용 연산 코어보다 훨씬 적은 전력으로 훨씬 많은 곱셈을 처리
- **Amdahl의 법칙**: 문제 크기를 고정한 채 컴퓨팅 자원을 아무리 늘려도, 병렬화가 안 되는 부분(직렬 구간)이 전체 속도 향상의 상한선을 정한다는 원리
- **강한 스케일링 vs 약한 스케일링**: 강한 스케일링은 "같은 크기의 작업을 자원만 늘려 더 빨리 끝내기", 약한 스케일링은 "자원을 늘린 만큼 더 큰 작업을 처리하기"
- **SM(스트리밍 멀티프로세서)과 워프(Warp)**: SM은 GPU 내부의 연산 단위 블록, 워프는 SM이 한 번에 같은 명령어로 실행하는 스레드 32개 묶음
- **PTX와 SASS**: PTX는 GPU 세대가 바뀌어도 그대로 쓸 수 있는 중간 단계 가상 명령어, SASS는 그 밑에서 칩마다 다르게 번역되는 실제 기계어(Nvidia가 상세 공개를 안 함)
- **MMA 명령어 (Multiply-Accumulate)**: "D = A×B+C" 형태의 행렬 곱셈+누적을 한 번에 처리하는 텐서 코어의 핵심 명령어
- **TMA / TMEM (텐서 메모리 가속기 / 텐서 전용 메모리)**: TMA는 대용량 데이터를 메모리 간에 자동으로 옮겨주는 전용 하드웨어, TMEM은 텐서 코어 연산 결과를 담아두는 Blackwell의 신설 전용 메모리 공간
- **구조적 희소성 (Structured Sparsity)**: 행렬 값 중 일정 비율을 0으로 만들어(가지치기) 계산량을 줄이는 기법 — 이론상 연산 속도를 2배 높일 수 있음

## 1. 서론: 텐서 코어를 알아야 하는 이유

**📌 핵심:**
- SemiAnalysis는 작년 "AI 스케일링 법칙" 리포트에서, AI 산업이 무어의 법칙보다 빠른 성능 향상을 이어온 배경에는 학습·추론 소프트웨어 최적화뿐 아니라 하드웨어(연산 능력) 발전도 있다고 짚었음 — 이번 리포트는 그 하드웨어 축, 특히 AI 연산의 심장인 텐서 코어(Tensor Core)를 집중 해부
- 텐서 코어는 2017년 Volta GPU에 처음 탑재된 이후 8년간 5세대(Volta→Turing→Ampere→Hopper→Blackwell)를 거치며 AI 학습·추론 성능의 핵심 동력이 되어 왔지만, 진화 속도가 워낙 빨라 숙련된 실무자조차 최신 세대의 변화를 따라가기 어려운 상태
- 이 리포트는 각 세대를 단순 스펙 비교가 아니라 "왜 그렇게 설계를 바꿨는가"라는 동기까지 설명하는 것이 목표 — Colfax Research, Stanford Hazy Research, Princeton, Together AI, Modal 등 CUDA 커널 전문가와 Nvidia CUDA 발명자 Ian Buck, GPU 아키텍처 총괄 Jonah Alben의 자문을 받아 작성
- 결론: 텐서 코어 아키텍처의 진화 논리를 이해하면, Nvidia가 왜 매 세대 압도적 성능 향상을 만들어내는지, 그리고 이 설계 철학이 향후 GPU 로드맵에 어떻게 이어질지 가늠할 수 있음

## 2. 성능 공학의 기본 원칙: Amdahl의 법칙과 데이터 이동

**📌 핵심:**
- Amdahl의 법칙: 문제 크기가 고정된 상태에서는 컴퓨팅 자원(GPU 코어 수 등)을 아무리 늘려도 속도 향상에 한계가 있음 — 병렬화가 안 되는 직렬 구간의 실행 시간이 전체 속도 향상의 상한선을 결정
- 강한 스케일링(같은 작업을 자원만 늘려 더 빨리 끝내기)과 약한 스케일링(자원을 늘린 만큼 더 큰 작업을 처리하기)은 서로 다른 성능 개선 곡선을 그림 — 이 구분이 이후 텐서 코어·CUDA 프로그래밍 모델 설계 전체를 관통하는 핵심 축
- 데이터 이동은 GPU 설계자들이 "죄악"이라 부를 만큼 비쌈 — 트랜지스터 연산은 1나노초 미만인데 D램 접근은 수십 나노초 걸려, 계산 자체보다 데이터를 옮기는 데 시간·에너지가 훨씬 많이 소모됨(메모리 벽)
- 결론: 이 세 가지 원칙이 이후 모든 텐서 코어 세대의 설계 변화(코어 크기 확대, 메모리 계층 신설, 비동기 실행 확대)를 관통하는 이론적 뼈대

---

강한 스케일링은 문제 크기와 무관하게 항상 속도가 빨라지지만, 약한 스케일링은 "자원을 늘린 만큼 더 큰 문제를 푼다"는 조건에서만 성능 향상이 보장됩니다.

**📌 용어 풀이: 데이터 이동이 "죄악"인 이유**
> - 연산(계산)은 저렴하고 데이터 이동은 비싸다는 것이 GPU 설계의 대전제 — 최신 트랜지스터는 1나노초도 안 걸려 스위칭하지만, D램 셀 접근에는 수십 나노초가 걸림
> - 2000년대 이후 연산 속도 향상은 계속 둔화됐지만, 메모리 속도 향상은 그보다도 더 느려 격차(메모리 벽)가 점점 벌어지는 중
> - 이 때문에 GPU 설계자들은 "얼마나 빨리 계산하는가"보다 "얼마나 데이터 이동을 줄이는가"에 갈수록 더 공을 들이게 됨 — 뒤에 나올 텐서 코어의 메모리 계층 신설(공유메모리·TMEM 등)이 모두 이 원칙에서 출발

## 3. 텐서 코어 이전: CUDA 프로그래밍 모델의 기초

**📌 핵심:**
- PTX(병렬 스레드 실행)는 GPU 세대가 바뀌어도 같은 프로그램이 그대로 동작하도록 해주는 가상 명령어 집합 — 실제 하드웨어에 종속되지 않는 중간 언어 역할
- GPU 하드웨어는 SM(스트리밍 멀티프로세서) 여러 개로 구성되고, 스레드 32개를 워프(Warp)라는 단위로 묶어 한 번에 같은 명령어를 실행하는 "SIMT" 방식으로 동작
- PTX 밑에는 SASS라는 아키텍처별 실제 명령어 집합이 있는데, Nvidia가 경쟁사에 설계 노하우를 숨기려고 상세 문서화를 거의 하지 않아 외부에 잘 알려지지 않음
- 결론: 이후 모든 텐서 코어 세대 변화는 이 PTX↔SASS 2계층 구조, 그리고 SM·워프·CTA(협력 스레드 배열)라는 스레드 위계 안에서 일어남 — 이 기본 틀을 알아야 세대별 차이가 왜 생겼는지 이해 가능

---

```mermaid
flowchart TD
    Kernel["커널 함수<br/>(GPU 스레드 대량 실행 프로그램)"] --> Grid["그리드(Grid)<br/>= CTA(협력 스레드 배열) 묶음"]
    Grid --> CTA["CTA 1개<br/>= 워프(Warp) 여러 개<br/>(워프 1개 = 스레드 32개)"]
    CTA --> Mem["메모리 3계층<br/>레지스터(스레드별)·<br/>공유메모리(CTA 내 공유)·<br/>글로벌메모리(전체 공유)"]

    style CTA fill:#fff7ed,stroke:#ea580c,stroke-width:2px
    style Mem fill:#eff6ff,stroke:#3b82f6
```

SM은 여러 개의 스칼라 연산 코어(CUDA 코어), 스레드를 관리하는 명령어 유닛, 칩 내장 공유 메모리로 구성됩니다. 명령어 유닛이 워프 하나를 골라 그 워프에 속한 32개 스레드 전체에 같은 명령어를 동시에 내리는 방식이 SIMT(단일 명령어, 다중 스레드)입니다.

```mermaid
flowchart TD
    PTX["PTX<br/>(가상 명령어 집합,<br/>세대와 무관하게 동작하는 중간 언어)"] --> SASS["SASS<br/>(아키텍처별 실제 명령어,<br/>Nvidia가 상세 문서화 거의 안 함)"]
    SASS --> HW["실제 GPU 하드웨어<br/>(SM, 텐서 코어 등)"]

    style PTX fill:#eff6ff,stroke:#3b82f6,stroke-width:2px
    style SASS fill:#fef2f2,stroke:#dc2626
```

**📌 용어 풀이: SIMT와 SIMD의 차이**
> - SIMD(단일 명령어, 다중 데이터)는 한 명령어로 여러 데이터를 동시에 처리하되, 벡터 폭(한 번에 몇 개를 처리할지)을 프로그래머가 직접 지정
> - SIMT(단일 명령어, 다중 스레드)는 벡터 폭 대신 "스레드 하나가 어떻게 동작하는지"만 지정하고, 나머지 병렬 처리는 하드웨어(워프 스케줄러)가 알아서 관리
> - 이 차이 덕분에 CUDA 프로그래밍은 SIMD보다 직관적인 "스레드 단위 사고"로 병렬 프로그램을 짤 수 있음

## 4. Volta: 텐서 코어의 탄생 (1세대)

**📌 핵심:**
- 2015년 Google이 자체 딥러닝 전용 칩(TPUv1)을 배치하며 업계에 충격을 준 이후, Nvidia도 2017년 행렬 연산 전용 하드웨어를 최초로 GPU에 탑재 — 개발 막바지(테이프아웃 몇 개월 전)에 급하게 추가될 만큼 시급한 결정이었음
- GPU가 명령어 하나를 처리하는 데 드는 전력(약 30pJ)이 단순 연산 자체(약 1.5pJ)보다 20배나 커서, 단순 연산을 잔뜩 반복하는 방식은 전력 낭비 — 명령어 1개로 훨씬 많은 연산을 처리하는 전용 명령어(HMMA)가 필요했음
- 이렇게 탄생한 1세대 텐서 코어는 V100 SM 1개당 8개 탑재, 코어 1개가 사이클당 4×4×4 행렬곱(SM 전체로는 사이클당 1,024 FLOPs)을 처리 — 스레드 8개(쿼드페어)가 함께 참여해 8×8×4 크기 연산을 수행
- 결론: 텐서 코어의 근본 존재 이유는 "명령어 오버헤드 대비 실제 연산량을 늘려 전력 효율을 높이는 것" — 이후 모든 세대의 진화(코어 크기 확대, 연산 참여 스레드 확대)가 이 방향의 연장선

---

```mermaid
flowchart TD
    Trend["2015년 Google TPUv1<br/>(딥러닝 전용 칩 최초 배치)"] --> Problem["문제: GPU 명령어 처리 전력이<br/>연산 자체보다 20배 큼<br/>(명령어 약 30pJ vs 연산 1.5pJ)"]
    Problem --> Solution["해법: HMMA<br/>(반정밀도 행렬곱+누산 명령어)<br/>명령어 1개로 연산량 극대화"]
    Solution --> V100["전용 하드웨어 탑재<br/>= 텐서 코어(2017년 V100, Volta)<br/>테이프아웃 몇 개월 전 긴급 추가"]

    style Problem fill:#fef2f2,stroke:#dc2626
    style V100 fill:#f0fdf4,stroke:#16a34a,stroke-width:2px
```

MMA(곱셈+누적) 명령어는 D = A×B+C 형태의 행렬 연산을 한 번에 수행합니다. A는 M×K, B는 K×N, C·D는 M×N 크기 행렬이며, 이 크기를 `MxNxK` 표기로 나타냅니다. 연산 순서는 다음과 같습니다.

- 공유메모리에서 A·B·C 행렬을 스레드 레지스터로 로드(각 스레드가 행렬의 일부 조각을 보유)
- MMA 명령어 실행 — 레지스터에서 값을 읽어 텐서 코어에서 계산 후 결과를 다시 레지스터에 저장
- 레지스터의 결과를 공유메모리로 다시 저장

이 전 과정은 여러 스레드가 협력해서 수행하므로, 매 단계마다 스레드 간 동기화가 필요합니다.

```mermaid
flowchart TD
    V100SM["Volta V100 SM 1개"] --> TC["텐서 코어 8개<br/>(2개씩 짝지어 4개 파티션)"]
    TC --> Perf["코어 1개당<br/>4×4×4 행렬곱/사이클<br/>SM 전체 1,024 FLOPs/사이클"]
    TC --> Thread["연산 참여 단위<br/>스레드 8개(쿼드페어)<br/>= 8×8×4 행렬곱 처리"]

    style TC fill:#fff7ed,stroke:#ea580c,stroke-width:2px
```

Volta 텐서 코어는 FP16(16비트 부동소수점) 입력을 FP32(32비트)로 누적하는 "혼합 정밀도" 방식을 지원합니다. 이 방식으로 정밀도를 낮춰도 모델 정확도 손실 없이 학습할 수 있음이 처음 입증됐습니다.

## 5. Turing: 2세대 텐서 코어와 정수 연산

**📌 핵심:**
- Turing(2018)은 텐서 코어 2세대 — Volta 대비 INT8·INT4 저정밀 정수 연산을 새로 지원해 추론(inference) 특화 성능을 강화
- 새로운 워프 단위 동기식 MMA(warp-level synchronous MMA) 방식을 도입 — 이후 3세대(Ampere)에서 본격 확장되는 "워프 전체가 함께 연산에 참여" 방식의 출발점
- 텐서 코어가 게임 그래픽에도 처음 활용돼 DLSS(딥러닝 슈퍼샘플링, 저해상도 이미지를 AI로 고해상도처럼 보이게 하는 기술)가 탄생 — AI 연산 하드웨어가 소비자 제품까지 확장된 시초
- 결론: Turing은 스펙 자체보다 "정수 연산 지원 확대"와 "워프 단위 MMA로의 전환 시작"이라는 방법론적 전환점으로서 의미가 큼

## 6. Ampere: 3세대 텐서 코어와 비동기 데이터 복사

**📌 핵심:**
- Ampere(2020)는 텐서 코어 3세대 — SM 1개당 코어 수는 4개로 Volta(8개)의 절반이지만, 코어 1개당 처리량(512 FLOPs/사이클)이 늘어 SM 전체로는 2,048 FLOPs/사이클(Volta 대비 2배)
- 연산 참여 단위가 스레드 8개(쿼드페어)에서 워프 전체(32개)로 확대 — 스레드 배치가 단순해지고 레지스터 압박도 줄어듦
- **비동기 데이터 복사** 신설 — 기존에는 데이터를 글로벌메모리→레지스터→공유메모리 2단계로 옮겨야 했는데, 이 과정이 MMA 연산과 레지스터를 두고 경합해 병목이 됐음. Ampere는 레지스터를 거치지 않고 글로벌메모리에서 공유메모리로 직접 데이터를 옮겨, 레지스터를 MMA 연산에 더 많이 쓸 수 있게 함
- 결론: Ampere는 순수 연산 성능(2배)뿐 아니라, 데이터 이동과 연산을 동시에 진행할 수 있는 비동기 실행의 첫걸음을 뗀 세대 — 이후 세대일수록 이 "비동기화" 흐름이 더 강화됨

---

```mermaid
flowchart TD
    OldWay["기존 방식(Volta)<br/>글로벌메모리→레지스터→공유메모리<br/>(2단계 경유)"] --> Problem["문제: MMA 연산도<br/>레지스터를 많이 써서<br/>데이터 로딩과 레지스터 경합<br/>(레지스터 압박 심화)"]
    Problem --> New["Ampere: 비동기 데이터 복사<br/>글로벌메모리→공유메모리<br/>직접 이동(레지스터 안 거침)"]
    New --> Benefit["레지스터를 MMA 연산에<br/>더 많이 활용 가능<br/>+ 데이터 로딩·연산 동시 진행"]

    style Problem fill:#fef2f2,stroke:#dc2626
    style New fill:#f0fdf4,stroke:#16a34a,stroke-width:2px
```

비동기 데이터 복사는 PTX 명령어 `cp.async`로 구현되며, 실제로는 SASS 명령어 `LDGSTS`(글로벌→공유메모리 비동기 복사)로 번역됩니다.

```mermaid
flowchart TD
    AmpereSM["Ampere SM 1개"] --> TC["텐서 코어 4개<br/>(코어 수는 Volta의 절반)"]
    TC --> Perf["코어 1개당 512 FLOPs/사이클<br/>SM 전체 2,048 FLOPs/사이클<br/>(Volta 대비 2배)"]
    TC --> Thread["연산 참여 단위<br/>워프 전체(32스레드)<br/>(Volta는 8스레드 쿼드페어)"]

    style Perf fill:#fff7ed,stroke:#ea580c,stroke-width:2px
```

Ampere는 `ldmatrix`(워프 단위로 여러 스레드가 함께 행렬을 로드하는 명령어)도 새로 도입했습니다. 개별 로드 명령어를 여러 번 내리는 대신 한 번에 처리해, 주소 계산용 레지스터 사용을 줄이고 프로그래밍도 단순해졌습니다.

**📌 용어 풀이: BF16이 대세가 된 이유**
> - FP32(32비트)는 지수 8비트+가수 23비트, FP16(16비트)은 지수 5비트+가수 10비트로 구성돼 표현 범위(다이나믹 레인지)가 FP32보다 훨씬 좁음
> - Ampere가 지원한 BF16은 지수 8비트(FP32와 동일)+가수 7비트로 설계돼, FP32 수준의 표현 범위를 절반 크기(16비트)로 담아냄
> - 표현 범위가 좁으면 학습 중 숫자가 너무 크거나 작아져 깨지는 문제가 잦은데, BF16은 FP32와 같은 범위를 써서 "로스 스케일링"이라는 별도 보정 기법 없이도 안정적으로 학습 가능 — 이후 BF16이 절반 정밀도의 사실상 표준으로 자리잡음
