# SemiAnalysis Newsletter Transformation Rules

이 문서는 SemiAnalysis 뉴스레터를 가공할 때 적용하는 변환 규칙을 정리합니다.

## 목표 독자
- 비즈니스 고위층 (경영진, 임원, 투자자)
- 기술 전문가가 아닌 비즈니스 전문가
- 빠른 의사결정이 필요한 리더

## 파일명 규칙
- **형식**: `[YYYY-MM-DD] 한글 제목.md`
- **예시**: `[2024-10-14] 데이터센터 해부학 Part 1 - 전기 시스템.md`
- **이유**: 날짜로 정렬 가능, 한글로 내용 파악 용이

## 문서 구조

### 1. 헤더 섹션
```markdown
# 영문 원제목

> **출처**: [링크]
> **저자**: 저자명
> **발행일**: YYYY-MM-DD
```

### 2. 목차 (📑)
- 클릭 가능한 앵커 링크
- 완료된 섹션 vs 작성 예정 구분
- 난이도 표시: 🟢 (쉬움) 🟡 (중간) 🔴 (어려움)

### 3. 본문 섹션 구조

각 주요 섹션은 다음 구조를 따름:

```markdown
## [난이도] 섹션 제목

**📌 핵심:**
- [4줄 이내 핵심 요약]
- [경영진이 즉시 이해 가능한 평이한 언어 사용]
- [숫자와 비유로 구체화]
- 결론: [핵심 메시지]

---

### 세부 내용 제목

설명 텍스트...

**🎯 이 다이어그램이 설명하는 것:**
- [4-5개 항목으로 다이어그램의 목적 설명]
- [무엇을 시각화하는지 명확히]
- 핵심: [이 다이어그램에서 가장 중요한 인사이트]

```mermaid
[다이어그램 코드]
```

**📌 용어 풀이: [전문 용어]**
> - **용어1**: 평이한 설명
> - **용어2**: 평이한 설명
> - **쉬운 비유**: 일상적 비유로 설명
```

## 핵심 원칙

### Option 2: 용어 풀이는 다이어그램 내부에 통합
- ✅ **채택**: 다이어그램 안에 용어 설명 포함
- ❌ **기각**: 다이어그램 전에 별도 용어 풀이 섹션

**이유**:
- 다이어그램과 설명이 분리되면 읽기 불편
- 다이어그램 안에 설명이 있으면 한눈에 이해 가능
- 필요한 경우 다이어그램 후 추가 용어 풀이 블록 사용

### 다이어그램 사용 원칙

1. **목적 명시 필수**
   - 모든 다이어그램 전에 "🎯 이 다이어그램이 설명하는 것:" 섹션 필수
   - "시각적으로 보여줍니다" 같은 모호한 표현 금지
   - 구체적으로 무엇을 설명하는지 명시

2. **다이어그램 내 용어 설명**
   - 노드 안에 용어를 한글과 함께 표시
   - 예: `"Direct-to-chip 액체냉각<br/>(물로 칩 직접 냉각)"`
   - 괄호 안에 평이한 설명 추가

3. **적절한 사용**
   - 복잡한 관계를 단순화할 때만 사용
   - 단순한 나열은 표나 목록으로 표현

## 평이한 언어 사용 규칙

### "📌 핵심:" 섹션 작성 규칙

**금지 용어 (전문 용어)**:
- ❌ 가용성 → ✅ 연간 1.6시간 다운타임
- ❌ Redundancy 전략 → ✅ 백업 장비 전략
- ❌ Hyperscale → ✅ 초대형 시설
- ❌ Colocation → ✅ 임대
- ❌ Capex → ✅ 초기 투자
- ❌ HV/MV/LV → ✅ 초고압/중압/저압
- ❌ Generator → ✅ 발전기
- ❌ UPS → ✅ 배터리
- ❌ Lead Time → ✅ 소요 시간
- ❌ direct to chip → ✅ 칩에 직접 물 공급
- ❌ 액체냉각 → ✅ 물 냉각 시스템
- ❌ 전력밀도 → ✅ 좁은 공간에 많은 전력

**작성 원칙**:
1. 숫자로 구체화 (예: "13배 많은 전력", "가정 130채 분량")
2. 일상적 비유 사용 (예: "가정 X채", "수백억원 손실")
3. 한 문장에 하나의 핵심만
4. 전문 용어 사용 시 즉시 풀이

### 본문 작성 규칙

**전문 용어 사용 시**:
- 처음 등장 시 즉시 설명
- 괄호나 박스로 명확히 구분
- 필요시 "📌 용어 풀이:" 블록 사용

**숫자 표현**:
- 추상적 단위 → 구체적 비유
- 예: "130kW" → "가정 130채 분량"
- 예: "300MW" → "미국 가정 20만 채 전력"

## Mermaid 다이어그램 스타일 가이드

### 색상 코딩
```
중요/경고: fill:#fff7ed,stroke:#ea580c
위험/문제: fill:#fef2f2,stroke:#dc2626
성공/긍정: fill:#f0fdf4,stroke:#16a34a
정보/중립: fill:#eff6ff,stroke:#3b82f6
강조: stroke-width:2px
```

### 노드 텍스트 포맷
- 한글 제목 + 영문 원문 (처음 등장 시)
- 괄호 안에 평이한 설명
- `<br/>` 태그로 줄바꿈
- 수치는 구체적으로 (예: "가정 130채 분량")

## 난이도 표시

- 🟢 **쉬움**: 기본 개념, 비전문가도 쉽게 이해
- 🟡 **중간**: 일부 기술 개념 포함, 설명과 함께 이해 가능
- 🔴 **어려움**: 복잡한 기술 개념, 집중 필요

## 진행 상황 표시

문서 끝에 진행 상황 표시:
```markdown
---

*작성 진행률: 약 XX% 완료*
*업데이트: [최근 업데이트 내용]*
```

## 체크리스트

각 섹션 완료 시 확인:
- [ ] 📌 핵심: 4줄 이내, 평이한 언어, 결론 포함
- [ ] 🎯 다이어그램 목적: 구체적 설명, "시각적으로" 금지
- [ ] 다이어그램: 용어 설명 포함, 적절한 색상
- [ ] 📌 용어 풀이: 필요시 블록 추가
- [ ] 난이도 표시: 🟢🟡🔴 적절히 배정
- [ ] 전문 용어: 모두 설명됨

## 예시

### 좋은 예시 ✅
```markdown
**📌 핵심:**
- Blackwell GB200은 기존 대비 **13배 많은 전력**(130kW, 가정 130채 분량)과 **물 냉각 시스템** 필수
- Meta는 건설 중 건물 철거 (수백억원 손실), Google은 즉시 배치 가능
- 결론: 대량 전력 공급 + 물 냉각 인프라 준비가 AI 시대 생존 조건
```

### 나쁜 예시 ❌
```markdown
**📌 핵심:**
- Blackwell GB200은 높은 전력밀도와 direct-to-chip 액체냉각 필요
- 기존 datacenter는 가용성 문제로 대응 불가
- Redundancy 전략 재검토 필요
```

**문제점**:
- 전력밀도, direct-to-chip, 액체냉각, 가용성, Redundancy 등 전문 용어 미설명
- 숫자로 구체화하지 않음
- 비즈니스 임팩트 불명확
