# SemiAnalysis Newsletter Transformation Rules

이 문서는 SemiAnalysis 뉴스레터를 가공할 때 적용하는 변환 규칙을 정리합니다.

## 목표 독자
- 비즈니스 고위층 (경영진, 임원, 투자자)
- 기술 전문가가 아닌 비즈니스 전문가
- 빠른 의사결정이 필요한 리더

## 파일명 규칙
- **형식**: `[YYMMDD] 한글 제목.md` (연도 2자리로 축약 — 파일명이 너무 길어지는 것 방지)
- **예시**: `[241014] 데이터센터 해부학 Part 1 - 전기 시스템.md`
- **이유**: 날짜로 정렬 가능, 한글로 내용 파악 용이, 2자리 연도로 파일명 단축
- 문서 내부 헤더의 "발행일" 표기는 그대로 `YYYY-MM-DD` 전체 표기 유지 (파일명만 단축, 본문 가독성은 유지)

## 문서 구조

문서는 아래 순서를 따릅니다. 각 부분의 상세 작성 규칙은 이어지는 섹션(본문 작성 규칙, 다이어그램 규칙 등)에서 다룹니다.

### 1. 헤더 섹션
```markdown
---
categories: [카테고리1, 카테고리2]
---

# 영문 원제목

> **출처**: [링크]
> **저자**: 저자명
> **발행일**: YYYY-MM-DD
```
- `categories`는 CATEGORIES.md에 정의된 유효 카테고리 중에서 선택 (여러 개 가능)
- 새 카테고리가 필요하면 임의로 만들지 말고 CATEGORIES.md를 먼저 갱신

### 2. 목차 (📑)
- 클릭 가능한 앵커 링크
- 완료된 섹션 vs 작성 예정 구분
- **신호등 이모지(🟢🟡🔴) 등 난이도 표시 금지** — 목차·섹션 헤더 모두 순수 텍스트 제목만 사용
- **번호 정렬**: 10개 이상 섹션이 있을 때 1-9 앞에 공백을 추가해 숫자 "."을 세로로 정렬 (가독성 향상)
  ```markdown
   1. [첫 번째 섹션]
   ...
   9. [아홉 번째 섹션]
  10. [열 번째 섹션]
  11. [열한 번째 섹션]
  ```

### 3. 용어 정리 (🔑) — 목차 직후, 본문 시작 전

**배치 이유**: 독자가 섹션을 순서대로 읽어 내려갈 때, 뒤쪽에서야 정의되는 용어가 앞쪽 문장에 먼저 등장하면 이해가 끊김. 목차 직후에 낯선 용어를 먼저 파악하게 하면 본문을 순서대로 읽어도 막히지 않음.

**포함 대상 (중요, v12.0/v13.1 변경)**: 목차(섹션·소제목)에 등장하는 어려운 용어로 한정 — 본문에 흩어진 "📌 용어 풀이:" 대상을 전부 끌어오지 않음. 일반 약어(IT, CPU, GPU, AI, API 등)는 애초에 제외.
- **기본 범위**: 목차의 섹션·소제목에 등장하는 전문 용어만 포함 (예: "데이터 홀과 Pod" 섹션이 있으면 Data Hall, Pod 포함)
- **보충 조건**: 위 기준으로 추린 용어 수가 **4개 이하**일 때만, 본문에서 자주 등장하거나 핵심적인 용어를 추가로 보충
- **상한 (v13.1)**: 최대 **8개**. 목차 기준으로 추린 용어가 8개를 넘으면, 가장 핵심적이고 본문에서 자주 반복되는 용어 위주로 8개까지만 추림 — 나머지는 목차 용어라도 글로서리에서 제외 (본문 "📌 용어 풀이:" 블록에서는 그대로 설명)
- **이유**: 본문의 모든 "📌 용어 풀이:" 대상을 다 끌어오면 목차 직후 용어 정리가 지나치게 길어져(문서 하나에 20개 이상까지 늘어난 사례 있음) 오히려 "목차 뒤 빠르게 훑어보기"라는 원래 목적을 해침 — 상한선을 둬서 4~8개 범위로 고정

**성격 (중요)**: 단순 약어 풀이(예: "UPS = Uninterruptible Power Supply")가 아니라, **그것이 실제로 무엇을 하는 것인지 개념을 설명**하는 것
- 예: "MVA vs MW"는 약어 전개가 아니라 "겉보기 전력 vs 실제 전력"이라는 개념 차이를 설명
- 예: "전압 레벨"은 HV/MV/LV 나열이 아니라 왜 단계별로 전압을 낮추는지, 각 단계가 무슨 역할을 하는지 설명

**분량 (중요): "이게 뭐다" 수준까지만**
- 용어당 한 줄, "이 용어가 무엇인지"만 설명 — 숫자, 스펙, 장단점 목록, 세부 예시는 넣지 않음
- ❌ 잘못된 예: "UPS: 응답시간 10ms, 위치는 변압기 다음, 배터리 종류는..." (수치·스펙 나열)
- ✅ 올바른 예: "UPS (무정전 전원 공급 장치): 정전 순간부터 발전기가 켜지기 전까지 배터리로 전력을 이어주는 장치"
- 구체적 수치, 비교, 장단점, 사례는 본문에만 작성 (본문의 "📌 용어 풀이:" 블록이나 관련 단락에서 상세히 다룸)

**본문과의 관계**: 목차 뒤 용어 정리는 개요용 한 줄 설명, 본문 해당 위치의 "📌 용어 풀이:" 블록은 상세 버전으로 그대로 유지 (역할 분담: 요약 vs 상세, 중복 허용)
- 다이어그램 내부 노드 텍스트의 짧은 병기 설명(예: `"UPS<br/>(무정전 전원 공급 장치)"`)도 그대로 유지

### 4. 본문 섹션 구조

각 주요 섹션은 다음 구조를 따름:

```markdown
## [번호]. 섹션 제목

**📌 핵심:**
- [4줄 이내 핵심 요약]
- [경영진이 즉시 이해 가능한 평이한 언어 사용]
- [숫자와 비유로 구체화]
- 결론: [핵심 메시지]

---

### 세부 내용 제목

설명 텍스트...

```mermaid
[다이어그램 코드]
```

**📌 용어 풀이: [전문 용어]**
> - **용어1**: 평이한 설명
> - **용어2**: 평이한 설명
> - **쉬운 비유**: 일상적 비유로 설명
```

작성 세부 규칙은 "본문 작성 규칙"과 "다이어그램 규칙" 참고.

## 본문 작성 규칙

### 평이한 언어 사용

**"📌 핵심:" 섹션 금지 용어 (전문 용어)**:
- ❌ 가용성 → ✅ 연간 1.6시간 다운타임
- ❌ Redundancy 전략 → ✅ 백업 장비 전략
- ❌ Hyperscale → ✅ 초대형 시설
- ❌ Colocation → ✅ 임대
- ❌ Capex → ✅ 초기 투자
- ❌ HV/MV/LV → ✅ 초고압/중압/저압
- ❌ Generator → ✅ 발전기
- ❌ UPS → ✅ 배터리
- ❌ Lead Time → ✅ 소요 시간
- ❌ direct to chip → ✅ 칩에 직접 물 공급
- ❌ 액체냉각 → ✅ 물 냉각 시스템
- ❌ 전력밀도 → ✅ 좁은 공간에 많은 전력

**작성 원칙**:
1. 숫자로 구체화 (예: "13배 많은 전력", "가정 130채 분량")
2. 일상적 비유 사용 (예: "가정 X채", "수백억원 손실")
3. 한 문장에 하나의 핵심만
4. 전문 용어 사용 시 즉시 풀이

**원문 전체 커버 (v16.0)**: 원문의 모든 섹션을 빠짐없이 변환한다. 분량을 줄이기 위해 원문 섹션을 통째로 생략하는 것은 금지.

**숫자 표현**:
- 추상적 단위 → 구체적 비유
- 예: "130kW" → "가정 130채 분량"
- 예: "300MW" → "미국 가정 20만 채 전력"

**환산 계수 실측 근거 (v16.0)**: "가정 X채 분량" 같은 비유 환산은 실제 통계(예: EIA 미국 평균 가구 전력 소비량)에 근거한 계수를 사용. 그럴듯한 어림수를 창작하지 않기.

**숫자 범위 표기 (v13.3)**: 물결표는 반드시 이스케이프해서 `\~` 로 표기 (예: `12\~24개월`, `$1,700\~2,000/kW`)
- **이유**: 마크다운 렌더러에 따라 이스케이프 없는 `~`가 취소선 문법으로 오인될 수 있음 — 특히 같은 문단·문장 안에 범위 표기(`~`)가 두 번 이상 나오면, 렌더러가 첫 `~`부터 다음 `~`까지를 취소선으로 처리해버리는 사례가 확인됨
- **적용 범위**: 본문·표·용어 정리 등 일반 마크다운 텍스트에서만 이스케이프. mermaid 코드 블록(` ```mermaid ` ~ ` ``` `) 안의 노드 텍스트는 마크다운이 해석되지 않으므로 이스케이프하지 않고 `~` 그대로 사용

### 약어 및 전문 용어 설명

**약어 분류**:
- **일반 약어 (설명 불필요)**: IT, CPU, GPU, AI, API 등 — 한국인 대부분이 아는 기술 용어, 설명 없이 그대로 사용
- **전문 약어 (설명 필요)**: UPS, BBU, OCP, PDU, ATS, STS 등 — 전문가가 아니면 모르는 용어, 반드시 설명 추가

**설명 위치와 형식**:
1. **목차 및 섹션 헤더**: 한국어 의미 설명만 포함
   - 예: `## UPS (무정전 전원 공급 장치) 시스템`
   - 예: `## OCP 랙과 BBU (배터리 예비 전원)`
2. **본문 첫 등장**: 영문 원어 + 한국어 설명
   - 예: `UPS (Uninterruptible Power Supply, 무정전 전원 공급 장치)`
   - 예: `BBU (Battery Backup Unit)`, `OCP (Open Compute Project)`
3. **이후 등장**: 약어만 사용 (예: `UPS`, `BBU`, `OCP`)

**한국어 번역 원칙**:
- 의미 있는 번역 사용: ✅ UPS → "무정전 전원 공급 장치" (의미 명확), ✅ BBU → "배터리 예비 전원"
- ❌ BBU → "배터리 백업 유닛" (단순 직역, 의미 없음)
- 번역이 의미 없는 경우 영문만: OCP → "오픈 컴퓨트 프로젝트" 대신 `OCP (Open Compute Project)`
- 억지로 번역하지 않기: "datacenter" → 문맥에 따라 "데이터센터" 또는 그대로, 자연스러운 것을 우선

### 논리적 연결고리 체크 (Logical Completeness)

**중요**: "📌 핵심:" 섹션은 독립적으로 읽어도 이해 가능해야 함

**체크 포인트**:
1. **인과관계 명시**
   - ❌ "GB200 필요 → Meta 건물 철거"
   - ✅ "GB200은 130kW 필요 → Meta 건물은 10kW만 지원 → 수용 불가 → 철거"
2. **숫자 비교 제공**
   - ❌ "전력이 부족해서"
   - ✅ "10kW vs 130kW (13배 부족)"
3. **결과의 이유 설명**
   - ❌ "Meta는 손실, Google은 즉시 배치"
   - ✅ "Meta(10kW)는 철거 손실, Google(40kW)은 즉시 배치"

**자가 점검 질문**: 각 핵심 포인트를 읽고 질문
- "왜?"라고 물었을 때 바로 답이 있는가?
- 앞 문장과 뒷 문장 사이에 빠진 단계는 없는가?
- 비전문가가 읽고 "무슨 소리야?"라고 할 부분은 없는가?

**예시**:

나쁜 예 ❌:
```markdown
- Blackwell GB200은 기존 대비 13배 많은 전력 필요
- Meta는 건설 중 건물 철거 (수백억원 손실)
```
문제: Meta가 왜 철거했는지 논리적 연결고리 없음

좋은 예 ✅:
```markdown
- Blackwell GB200은 13배 많은 전력(130kW) 필요
- Meta 건물은 10kW만 지원 → GB200 수용 불가 → 철거 결정 (수백억원 손실)
- Google은 40kW 지원 → 즉시 배치 가능
```
장점: 왜 철거했는지 → 10kW로는 130kW 수용 불가능하기 때문

### 비교 표현 규칙

**비교 대상 명시 필수**

나쁜 예 ❌:
```markdown
- GB200은 13배 많은 전력 필요
```
문제: 무엇과 비교해서 13배인지 불명확 (Hopper? 일반 랙?)

좋은 예 ✅:
```markdown
- GB200은 일반 랙 대비 13배 많은 전력 (일반 랙 10kW → GB200 130kW)
```
장점: 비교 대상과 구체적 수치 모두 명시

**체크리스트**:
- [ ] 비교 대상이 명확한가? (무엇과 비교?)
- [ ] 구체적 수치가 있는가? (얼마에서 얼마로?)
- [ ] 배수 표현의 근거가 있는가? (왜 X배인가?)

## 다이어그램 규칙

### 우선순위: 줄글보다 다이어그램 먼저 (Option 2 원칙, v11.0 복원)
- 개념·관계·인과관계를 설명할 때는 별도 줄글 단락보다 mermaid 다이어그램을 우선 사용 — 설명을 다이어그램 노드·화살표·라벨 안에 최대한 담고, 줄글은 다이어그램으로 담기 어려운 서사(사례, 인용, 배경)에만 씀
- 노드 텍스트 안에 용어의 한글 설명을 병기해 별도의 "📌 용어 풀이:" 단락 없이도 다이어그램만으로 이해되게 작성 (아래 "노드 텍스트 포맷" 참고)
- 판단 기준: "이 단락을 다이어그램으로 바꾸면 더 짧고 명확해지는가?" → 그렇다면 다이어그램으로 전환
- 목차 직후 용어 정리(🔑) 섹션은 그대로 유지 — 다이어그램 우선 원칙과 별개로, 본문을 순서대로 읽기 전 낯선 용어를 미리 파악하는 용도
- **정량 기준 (v17.2, 글자수 기준)**: "순수 줄글"(mermaid 블록 제외, 헤더·불릿·인용·표·코드가 아닌 일반 문단 줄)의 **글자수 합이 파일 전체 글자수의 30% 이하**여야 함 — 기준작 Part 1(전기 시스템)이 15%, 양호한 문서들이 15\~27% 수준. 목표는 20% 내외
- **문단 줄 길이 상한 (v17.2)**: 한 문단 줄은 **200자 이하** — 여러 문장을 한 줄에 이어붙여 줄 수를 줄이는 편법 방지(기준작 Part 1의 최장 문단 줄은 181자). 200자를 넘는 문단은 나눌 게 아니라 다이어그램 전환 후보
- (참고) v17.1의 "줄 수 10% 이하" 기준은 폐기 — 줄 수 기준은 문장을 한 줄에 합쳐 쓰는 방식으로 우회 가능함이 실제 사례로 확인됨 (메모리 마니아: 줄 비율 9%로 통과했으나 문단 평균 257자, 글자수 비율 41%)

### 다이어그램 추가 후 주변 줄글 중복 제거 (필수, v15.0)
- 표·줄글을 다이어그램으로 바꾸거나 새 다이어그램을 추가한 뒤에는, **다이어그램 바로 앞뒤 문단이 다이어그램과 같은 숫자·같은 흐름을 다시 설명하고 있지 않은지 반드시 확인**하고, 중복되면 줄글 쪽을 줄이거나 삭제 — 다이어그램만 남기는 게 원칙
- 예외(중복 허용): "📌 핵심:" 요약 블록(문서 맨 위 요약이라 본문과 겹치는 게 정상), "📌 용어 풀이:" 블록(다이어그램엔 없는 상세 설명 담당), 다이어그램을 소개하는 짧은 전환 문장("아래 표는 ~을 보여줍니다" 류)
- 실수 사례: 표를 다이어그램으로 바꾸는 작업을 여러 차례 거치면서, 원래 있던 설명 문단은 그대로 둔 채 다이어그램만 추가로 끼워 넣어 같은 내용이 문단+다이어그램 두 번 나오는 경우가 다수 발견됨(예: "① Critical IT Power ② 가동률 ③ PUE"를 문단으로 다 설명한 직후 같은 3단계를 다이어그램으로 또 보여줌)

### 적절한 사용 (v13.2: 표 사용을 예외로 축소)
- **기본값은 다이어그램**: 스펙 비교, 기술 A vs B, 수치 나열, 단계별 흐름 등 거의 모든 내용을 다이어그램으로 표현 — Part 1(전기 시스템, 855줄 중 표 단 1개)이 기준
- **표는 예외적으로만**: 정말로 다이어그램으로 바꿔도 이득이 없는 경우(예: 3개 카테고리 x 5개 속성처럼 순수 매트릭스형 분류표)에 한해 표 허용
- ❌ 표로 처리하면 안 되는 것(다이어그램으로): 두 가지 이상 기술·옵션의 트레이드오프 비교, 원인→결과로 이어지는 수치, 시계열 변화, 조직·기관 간 관계, 의사결정 분기
- ✅ 표가 허용되는 것: 성격이 다른 항목들을 순수하게 나란히 분류만 하는 매트릭스 (예: Retail/Wholesale/Hyperscale처럼 서로 인과관계 없는 카테고리 속성 비교)
- 판단 기준: "이 표를 다이어그램 노드·화살표로 바꾸면 관계나 흐름이 더 잘 보이는가?" → 그렇다면 다이어그램으로 전환

### 목적 명시 필수
- 다이어그램 전에 무엇을 설명하는지 구체적으로 명시
- "시각적으로 보여줍니다" 같은 모호한 표현 금지

### 방향: 항상 상하(TD)로만
- ✅ **채택**: `flowchart TD` / `graph TD` (위→아래)
- ❌ **기각**: `graph LR` / `flowchart LR` (좌→우)
- **이유**: 모바일로 읽는 독자가 많음. 좌우 방향 다이어그램은 화면 폭을 넘어가 잘리거나 가로 스크롤이 필요해 사실상 안 보임. 위→아래는 세로로 긴 모바일 화면에 자연스럽게 맞음

### 폭 제한: 같은 레벨 형제 노드는 최대 3개 (v14.0)
- **이유**: TD(상하) 방향이어도, 한 노드에서 자식 가지가 옆으로 여러 개 뻗으면 그 레벨의 가로 폭이 넓어져 모바일 화면을 넘어감 — 방향만 TD로 고정한다고 폭 문제가 해결되지 않음
- **직접 자식 노드 상한**: 한 노드에서 뻗는 가지는 최대 **3개**까지
- **손자 노드까지 포함한 폭 상한**: 자식 노드들이 각각 또 가지를 쳐서, 어느 한 레벨의 노드 수(그 레벨에 동시에 나열되는 손자 노드 총합)가 **4개 노드 폭 이상**이 되면 폭 초과로 간주
- **폭 초과 시 대응**: 그 레벨에서 노드를 줄이는 게 아니라, **부모 노드부터 구조를 다시 설계** — 예를 들어 (a) 다이어그램을 2개의 완전히 별도인 ` ```mermaid ` 코드 블록으로 쪼개기(같은 다이어그램 안에서 그룹으로 묶는 게 아니라, 아예 다른 코드 블록으로 분리), (b) 형제 관계를 순차적 흐름으로 바꿔 세로로 쌓기(단, 실제로 병렬적 형제 관계인데 순차 관계처럼 보이게 되는 부작용이 있으니 의미가 실제로 순서/단계일 때만 사용), (c) 여러 항목을 노드 하나의 텍스트 안에 글머리표(·)로 합쳐서 실제 노드 개수 자체를 줄이기
- **주의: 그룹 노드로 감싸는 것만으로는 부족함** — 같은 다이어그램 안에서 "부모 → 그룹 A/B/C → 각 그룹의 자식들" 식으로 중간 그룹 노드를 추가해도, mermaid는 랭크(rank, 루트로부터의 깊이) 기준으로 레이아웃을 계산하기 때문에 부모가 다른 노드라도 같은 깊이에 있으면 결국 한 줄에 나란히 배치됨 — 그룹 A/B/C의 자식을 다 합치면 여전히 4개 이상이면 폭 초과. 진짜 해결책은 (a) 완전히 별도의 코드 블록으로 분리하거나, (c)처럼 각 그룹의 여러 항목을 별도 자식 노드가 아니라 그룹 노드 자신의 텍스트 안에 합쳐서, 그 깊이에 존재하는 노드 수 자체를 줄이는 것
- **체크 방법**: 다이어그램을 다 그린 뒤 "이 레벨(깊이)에 몇 개 노드가 있는가?"를 각 depth마다 세어보고, 어느 한 레벨이라도 4개 이상이면 재설계

### 노드 텍스트 포맷
- 노드 안에 용어를 한글과 함께 표시 (한글 제목 + 영문 원문, 처음 등장 시)
- 괄호 안에 평이한 설명 추가 (예: `"Direct-to-chip 액체냉각<br/>(물로 칩 직접 냉각)"`)
- `<br/>` 태그로 줄바꿈
- 수치는 구체적으로 (예: "가정 130채 분량")

## 완료 후 검증 (필수)

문서 전체 작성이 끝나면(진행률을 100%로 올리기 전) 다음을 실행한다. 기계적으로 잡을 수 있는 위반은 grep 등으로 실제 명령을 돌려 확인한다:

1. **난이도 이모지 검사**: `grep -n "🟢\|🟡\|🔴" "파일명"` → 0건이어야 함
2. **다이어그램 방향 검사**: `grep -n "graph LR\|flowchart LR" "파일명"` → 0건이어야 함
3. **비이스케이프 물결표 검사**: mermaid 코드 블록을 제외한 본문에서 `\` 없이 쓰인 `~`를 검색 → 0건이어야 함. 예시 방법: python으로 ` ```mermaid ` \~ ` ``` ` 코드 블록을 제거한 뒤 정규식 `(?<!\\)~`로 검색
4. **다이어그램 폭 검사 (수동)**: 각 mermaid 블록에서 depth(랭크)별 노드 수를 세어 4개 이상인 레벨이 없는지 확인
5. **다이어그램-줄글 중복 검사 (수동, v15.0 규칙)**: 각 다이어그램 바로 앞뒤 문단을 다시 읽고 다이어그램과 같은 숫자·흐름을 반복하면 줄글 삭제
6. **목차-본문 대조**: 목차의 섹션 번호·제목이 본문 `## N. 제목`과 일치하고 앵커 링크가 유효한지 확인
7. **줄글 비율 검사 (v17.2, 글자수 기준)**: 아래 스크립트로 측정 → **문단 글자수 비율 30% 이하** 그리고 **200자 초과 문단 줄 0개**여야 함. 초과하면 가장 긴 문단부터 다이어그램으로 전환하거나 삭제 후 재측정 (줄 수를 세는 게 아니라 글자수를 세므로, 문장을 한 줄로 합치는 것은 해결이 아님):
   ```bash
   PYTHONIOENCODING=utf-8 python3 -c "
   import re
   text = open('파일명', encoding='utf-8').read()
   body = text
   for b in re.findall(r'\`\`\`mermaid.*?\`\`\`', body, re.S): body = body.replace(b, '')
   para = [l for l in body.split('\n') if l.strip() and not l.strip().startswith(('#','>','-','|','*','\`','---','['))]
   pc, tc = sum(map(len, para)), len(text)
   long = [l for l in para if len(l) > 200]
   print(f'문단글자 {pc} / 전체 {tc} = {pc*100//tc}% (기준 30% 이하) | 200자 초과 문단줄 {len(long)}개 (기준 0개)')
   "
   ```

## 완료 후 데이터 추출 (필수, v17.0)

문서 변환이 끝나면(검증 통과 후, 후속 갱신 전) 방금 읽은 원문 내용을 기억하고 있는 김에 아래 두 가지를 추출해 `대장/`의 두 파일에 추가한다. 대장 생성을 위해 나중에 문서를 다시 읽는 이중 작업을 피하는 것이 목적이므로, 반드시 변환과 같은 작업 안에서 수행한다:

1. **예측 → `대장/예측 대장.md`**: 시점(연도·분기·기한)이 박힌 예측을 전부 추출 (예: "2027년 HBM4 양산", "2028년까지 40GW 자가발전"). 저자 예측인지 업체 발표 인용인지 구분해 기록. 검증 상태 초기값은 ⏳ 시점 미도래.
2. **핵심 숫자 → `대장/숫자 대장.md`**: 문서 간 비교·충돌 검출 가치가 있는 정량 수치를 문서당 5\~15개 추출 — 단가($/kW, $/GPU), 밀도(kW/랙), 용량(GW/MW), 효율(%), TCO, 시장 규모, 점유율 등. 일회성 서사용 숫자(일화 속 금액 등)는 제외.

- 형식·컬럼은 각 대장 파일 상단의 사용 안내를 따름 (문서 단위 섹션으로 append)
- 이 추출까지 마쳐야 아래 후속 갱신으로 넘어갈 수 있음

## 완료 후 후속 갱신 (필수)

문서 완료는 번역 완료가 아니라 아래 후속 갱신까지 끝나야 완료로 간주한다:

1. **CATEGORIES.md 문서 수 표 갱신**: 이 문서의 frontmatter에 있는 **모든** 카테고리의 카운트를 +1 (여러 카테고리면 전부 — 일부만 갱신하는 실수 주의)
2. **통합 리포트 갱신**: 이 문서의 카테고리 중 `통합 리포트/카테고리별 통합 리포트/`에 이미 리포트가 있는 카테고리가 있으면, REPORT_RULES.md 트리거 2에 따라 그 리포트도 같은 작업 안에서 갱신 (리포트가 없는 카테고리는 새로 만들지 않음)
3. 위 갱신까지 마친 뒤에 진행률 footer를 100%로 설정하고 최종 커밋

**배경**: 800VDC 문서 처리 때 이 두 가지가 누락된 실제 사례가 있어 규칙화함.
