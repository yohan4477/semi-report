# SemiAnalysis Newsletter Transformation Rules

이 문서는 SemiAnalysis 뉴스레터를 가공할 때 적용하는 변환 규칙을 정리합니다.

## 목표 독자
- 비즈니스 고위층 (경영진, 임원, 투자자)
- 기술 전문가가 아닌 비즈니스 전문가
- 빠른 의사결정이 필요한 리더

## 파일명 규칙
- **형식**: `[YYYY-MM-DD] 한글 제목.md`
- **예시**: `[2024-10-14] 데이터센터 해부학 Part 1 - 전기 시스템.md`
- **이유**: 날짜로 정렬 가능, 한글로 내용 파악 용이

## 문서 구조

### 1. 헤더 섹션
```markdown
# 영문 원제목

> **출처**: [링크]
> **저자**: 저자명
> **발행일**: YYYY-MM-DD
```

### 2. 목차 (📑)
- 클릭 가능한 앵커 링크
- 완료된 섹션 vs 작성 예정 구분
- 난이도 표시: 🟢 (쉬움) 🟡 (중간) 🔴 (어려움)

### 3. 용어 정리 (🔑) — 목차 직후, 본문 시작 전
- 문서 전체에서 사용되는, 독자가 알기 어려운 용어를 모아서 설명
- 자세한 내용은 "핵심 원칙 > 문서 구조: 용어 정리는 목차 직후에 배치" 참고

### 4. 본문 섹션 구조

각 주요 섹션은 다음 구조를 따름:

```markdown
## [난이도] 섹션 제목

**📌 핵심:**
- [4줄 이내 핵심 요약]
- [경영진이 즉시 이해 가능한 평이한 언어 사용]
- [숫자와 비유로 구체화]
- 결론: [핵심 메시지]

---

### 세부 내용 제목

설명 텍스트...

**🎯 이 다이어그램이 설명하는 것:**
- [4-5개 항목으로 다이어그램의 목적 설명]
- [무엇을 시각화하는지 명확히]
- 핵심: [이 다이어그램에서 가장 중요한 인사이트]

```mermaid
[다이어그램 코드]
```

**📌 용어 풀이: [전문 용어]**
> - **용어1**: 평이한 설명
> - **용어2**: 평이한 설명
> - **쉬운 비유**: 일상적 비유로 설명
```

## 핵심 원칙

### 문서 구조: 용어 정리는 목차 직후에 배치
- ✅ **채택**: "🔑 용어 정리" 섹션을 목차(📑) 바로 다음, 본문 시작 전에 배치
- ❌ **기각**: 용어 설명을 본문 곳곳에만 흩어놓고 앞부분에 정리 없음

**이유**:
- 독자가 섹션을 순서대로 읽어 내려갈 때, 뒤쪽에서야 정의되는 용어가 앞쪽 문장에 먼저 등장하면 이해가 끊김
- 목차 직후에 낯선 용어를 먼저 파악하게 하면, 본문을 순서대로 읽어도 막히지 않음
- 독자가 알기 어려운 용어(전문 약어, 업계 용어)만 포함. 일반 약어(IT, CPU, GPU, AI, API 등)는 제외

**용어 정리의 성격 (중요)**:
- 단순 약어 풀이(예: "UPS = Uninterruptible Power Supply")가 아니라, **그것이 실제로 무엇을 하는 것인지 개념을 설명**하는 것
- 예: "MVA vs MW"는 약어 전개가 아니라 "겉보기 전력 vs 실제 전력"이라는 개념 차이를 설명
- 예: "전압 레벨"은 HV/MV/LV 약어 나열이 아니라 왜 단계별로 전압을 낮추는지, 각 단계가 무슨 역할을 하는지 설명
- 숫자·비유를 활용해 비전문가도 이해 가능하게 작성 (평이한 언어 사용 규칙과 동일한 기준 적용)

**본문과의 관계**:
- 목차 뒤 용어 정리 섹션에 모은 뒤에도, 본문 해당 위치의 "📌 용어 풀이:" 블록은 그대로 유지 (문맥 속에서 다시 확인 가능하도록)
- 다이어그램 내부 노드 텍스트의 짧은 병기 설명(예: `"UPS<br/>(무정전 전원 공급 장치)"`)도 그대로 유지

### 다이어그램 사용 원칙

1. **목적 명시 필수**
   - 모든 다이어그램 전에 "🎯 이 다이어그램이 설명하는 것:" 섹션 필수
   - "시각적으로 보여줍니다" 같은 모호한 표현 금지
   - 구체적으로 무엇을 설명하는지 명시

2. **다이어그램 내 용어 설명**
   - 노드 안에 용어를 한글과 함께 표시
   - 예: `"Direct-to-chip 액체냉각<br/>(물로 칩 직접 냉각)"`
   - 괄호 안에 평이한 설명 추가

3. **적절한 사용**
   - 복잡한 관계를 단순화할 때만 사용
   - 단순한 나열은 표나 목록으로 표현

## 논리적 연결고리 체크 (Logical Completeness)

**중요**: "📌 핵심:" 섹션은 독립적으로 읽어도 이해 가능해야 함

### 체크 포인트

1. **인과관계 명시**
   - ❌ "GB200 필요 → Meta 건물 철거"
   - ✅ "GB200은 130kW 필요 → Meta 건물은 10kW만 지원 → 수용 불가 → 철거"

2. **숫자 비교 제공**
   - ❌ "전력이 부족해서"
   - ✅ "10kW vs 130kW (13배 부족)"

3. **결과의 이유 설명**
   - ❌ "Meta는 손실, Google은 즉시 배치"
   - ✅ "Meta(10kW)는 철거 손실, Google(40kW)은 즉시 배치"

### 자가 점검 질문

각 핵심 포인트를 읽고 질문:
- "왜?"라고 물었을 때 바로 답이 있는가?
- 앞 문장과 뒷 문장 사이에 빠진 단계는 없는가?
- 비전문가가 읽고 "무슨 소리야?"라고 할 부분은 없는가?

### 예시

**나쁜 예** ❌:
```markdown
- Blackwell GB200은 기존 대비 13배 많은 전력 필요
- Meta는 건설 중 건물 철거 (수백억원 손실)
```
**문제**: Meta가 왜 철거했는지 논리적 연결고리 없음

**좋은 예** ✅:
```markdown
- Blackwell GB200은 13배 많은 전력(130kW) 필요
- Meta 건물은 10kW만 지원 → GB200 수용 불가 → 철거 결정 (수백억원 손실)
- Google은 40kW 지원 → 즉시 배치 가능
```
**장점**: 왜 철거했는지 → 10kW로는 130kW 수용 불가능하기 때문

## 평이한 언어 사용 규칙

### "📌 핵심:" 섹션 작성 규칙

**금지 용어 (전문 용어)**:
- ❌ 가용성 → ✅ 연간 1.6시간 다운타임
- ❌ Redundancy 전략 → ✅ 백업 장비 전략
- ❌ Hyperscale → ✅ 초대형 시설
- ❌ Colocation → ✅ 임대
- ❌ Capex → ✅ 초기 투자
- ❌ HV/MV/LV → ✅ 초고압/중압/저압
- ❌ Generator → ✅ 발전기
- ❌ UPS → ✅ 배터리
- ❌ Lead Time → ✅ 소요 시간
- ❌ direct to chip → ✅ 칩에 직접 물 공급
- ❌ 액체냉각 → ✅ 물 냉각 시스템
- ❌ 전력밀도 → ✅ 좁은 공간에 많은 전력

**작성 원칙**:
1. 숫자로 구체화 (예: "13배 많은 전력", "가정 130채 분량")
2. 일상적 비유 사용 (예: "가정 X채", "수백억원 손실")
3. 한 문장에 하나의 핵심만
4. 전문 용어 사용 시 즉시 풀이

### 본문 작성 규칙

**전문 용어 사용 시**:
- 처음 등장 시 즉시 설명
- 괄호나 박스로 명확히 구분
- 필요시 "📌 용어 풀이:" 블록 사용

**숫자 표현**:
- 추상적 단위 → 구체적 비유
- 예: "130kW" → "가정 130채 분량"
- 예: "300MW" → "미국 가정 20만 채 전력"

## 약어 및 전문 용어 설명 규칙

### 약어 분류

**일반 약어 (설명 불필요)**:
- IT, CPU, GPU, AI, API 등
- 한국인 대부분이 아는 기술 용어
- 설명 없이 그대로 사용

**전문 약어 (설명 필요)**:
- UPS, BBU, OCP, PDU, ATS, STS 등
- 전문가가 아니면 모르는 용어
- 반드시 설명 추가

### 약어 설명 위치와 형식

**1. 목차 및 섹션 헤더**:
- 한국어 의미 설명만 포함
- 예: `## UPS (무정전 전원 공급 장치) 시스템`
- 예: `## OCP 랙과 BBU (배터리 예비 전원)`

**2. 본문 첫 등장**:
- 영문 원어 + 한국어 설명
- 예: `UPS (Uninterruptible Power Supply, 무정전 전원 공급 장치)`
- 예: `BBU (Battery Backup Unit)`
- 예: `OCP (Open Compute Project)`

**3. 이후 등장**:
- 약어만 사용
- 예: `UPS`, `BBU`, `OCP`

### 한국어 번역 원칙

**의미 있는 번역 사용**:
- ✅ UPS → "무정전 전원 공급 장치" (의미 명확)
- ✅ BBU → "배터리 예비 전원" (의미 명확)
- ❌ BBU → "배터리 백업 유닛" (단순 직역, 의미 없음)

**번역이 의미 없는 경우**:
- OCP → "오픈 컴퓨트 프로젝트" (한국어로 해도 의미 불명확)
- 이런 경우 영문만 사용: `OCP (Open Compute Project)`

**억지로 번역하지 않기**:
- "datacenter" → 문맥에 따라 "데이터센터" 또는 그대로 사용
- 자연스러운 것을 우선

## 비교 표현 규칙

### 비교 대상 명시 필수

**나쁜 예** ❌:
```markdown
- GB200은 13배 많은 전력 필요
```
**문제**: 무엇과 비교해서 13배인지 불명확 (Hopper? 일반 랙?)

**좋은 예** ✅:
```markdown
- GB200은 일반 랙 대비 13배 많은 전력 (일반 랙 10kW → GB200 130kW)
```
**장점**: 비교 대상과 구체적 수치 모두 명시

### 비교 표현 체크리스트

- [ ] 비교 대상이 명확한가? (무엇과 비교?)
- [ ] 구체적 수치가 있는가? (얼마에서 얼마로?)
- [ ] 배수 표현의 근거가 있는가? (왜 X배인가?)

## 목차 형식 규칙

### 번호 정렬

**10개 이상 섹션이 있을 때**:
- 1-9 앞에 공백 추가하여 정렬
```markdown
 1. [첫 번째 섹션]
 2. [두 번째 섹션]
 ...
 9. [아홉 번째 섹션]
10. [열 번째 섹션]
11. [열한 번째 섹션]
```

**이유**: 숫자의 "."이 같은 세로 위치에 정렬되어 가독성 향상

## Mermaid 다이어그램 스타일 가이드

### 색상 코딩
```
중요/경고: fill:#fff7ed,stroke:#ea580c
위험/문제: fill:#fef2f2,stroke:#dc2626
성공/긍정: fill:#f0fdf4,stroke:#16a34a
정보/중립: fill:#eff6ff,stroke:#3b82f6
강조: stroke-width:2px
```

### 노드 텍스트 포맷
- 한글 제목 + 영문 원문 (처음 등장 시)
- 괄호 안에 평이한 설명
- `<br/>` 태그로 줄바꿈
- 수치는 구체적으로 (예: "가정 130채 분량")

## 난이도 표시

- 🟢 **쉬움**: 기본 개념, 비전문가도 쉽게 이해
- 🟡 **중간**: 일부 기술 개념 포함, 설명과 함께 이해 가능
- 🔴 **어려움**: 복잡한 기술 개념, 집중 필요

## 진행 상황 표시

문서 끝에 진행 상황 표시:
```markdown
---

*작성 진행률: 약 XX% 완료*
*업데이트: [최근 업데이트 내용]*
```

## 체크리스트

각 섹션 완료 시 확인:
- [ ] 📌 핵심: 4줄 이내, 평이한 언어, 결론 포함
- [ ] 🎯 다이어그램 목적: 구체적 설명, "시각적으로" 금지
- [ ] 다이어그램: 용어 설명 포함, 적절한 색상
- [ ] 📌 용어 풀이: 필요시 블록 추가
- [ ] 난이도 표시: 🟢🟡🔴 적절히 배정
- [ ] 전문 용어: 모두 설명됨

## 예시

### 좋은 예시 ✅
```markdown
**📌 핵심:**
- Blackwell GB200은 일반 랙 대비 **13배 많은 전력**(일반 랙 10kW → GB200 130kW, 가정 130채 분량)과 **물 냉각 시스템** 필수
- Meta는 건설 중 건물 철거 (수백억원 손실), Google은 즉시 배치 가능
- 결론: 대량 전력 공급 + 물 냉각 인프라 준비가 AI 시대 생존 조건
```

### 나쁜 예시 ❌
```markdown
**📌 핵심:**
- Blackwell GB200은 높은 전력밀도와 direct-to-chip 액체냉각 필요
- 기존 datacenter는 가용성 문제로 대응 불가
- Redundancy 전략 재검토 필요
```

**문제점**:
- 전력밀도, direct-to-chip, 액체냉각, 가용성, Redundancy 등 전문 용어 미설명
- 숫자로 구체화하지 않음
- 비즈니스 임팩트 불명확

  - 본문 내 기존 "📌 용어 풀이:" 블록과 다이어그램 내 병기 설명은 유지 (중복 허용)
  - 별도 "핵심정리" 요약 디렉토리/파일 생성 관행 폐기 (원본 해석 문서와 내용이 중복되어 유지보수 부담만 늘림)
